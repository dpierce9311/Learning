The Core Principle

Use "select" when the element's state changes and persists.
If the UI element becomes marked, chosen, or highlighted AND remains that way into the next step or section, use "select." The key question: Does it stay selected?
Use "click" for immediate actions without persistent state.
If clicking the element triggers an immediate action, opens something, or navigates somewhere WITHOUT the element itself maintaining a selected state, use "click."

Industry Standards & Your Rule

Your team's approach aligns with major technical writing standards. Here's what the research shows:

Use SELECT for:

  • Checkboxes (they stay checked)
  • Radio buttons (they stay selected)
  • Dropdown options (selection persists)
  • List items that remain highlighted
  • Tabs that show active state
  • Text/cells you're marking for an action

Use CLICK for:

  • Action buttons (Submit, Save, Cancel)
  • Links that navigate
  • Icons that open menus/dialogs
  • Buttons that perform immediate tasks
  • Elements with no persistent state change
  • Commands that execute instantly

Why This Matters

Consistent terminology improves:

  • Clarity: Users understand exactly what will happen
  • Scannability: "Select" signals a choice that persists
  • Accessibility: Screen readers handle these actions differently
  • Translation: Consistent verbs translate more reliably

Summary: The Decision Tree

  1. Is it a checkbox, radio button, or dropdown? → Use SELECT
  2. Does the element visually show it's selected/chosen? → Use SELECT
  3. Does the selection persist to the next step? → Use SELECT
  4. Is it an action button, link, or icon? → Use CLICK
  5. Does it immediately execute/navigate/open? → Use CLICK
