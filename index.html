1 Writing Effective Overview Sections

The overview is your user's first impression. It helps them quickly understand what the guide covers and confirms they've found the right resource for their task. A well-crafted overview sets clear expectations about what they'll learn.

üìñ Core Rules for Overviews

  • Single Process Guides (1-2 processes): Start with "This guide covers..." followed by a brief summary
  • Multi-Process Guides (3+ processes): Use "This guide covers business processes related to..." followed by the unifying theme
  • Business Process Format: Business Process Name (bold): Description starting with an action verb

2 Writing the Considerations Section

The Considerations section is for unique, process-specific information. Think of it as essential "need to know before you start" content that applies specifically to the business processes in your guide.

‚ö†Ô∏è What Belongs in Considerations

  • ‚úì System-specific behaviors unique to this process
  • ‚úì Dependencies that must be met before starting
  • ‚úì Conditions that affect how the process works
  • ‚úì Important warnings about irreversible actions
  • ‚úì Special permissions or roles required

‚úó What Does NOT Belong

  • ‚úó General Workday navigation (how to use prompts, required fields)
  • ‚úó Date entry formats (assumed knowledge)
  • ‚úó How to use selector buttons (universal Workday behavior)
  • ‚úó Basic dropdown usage (common across all processes)
  • ‚úó Standard field interactions (applies everywhere)

‚ùå Too General ‚Äì Don't Include

  • Required fields are marked with an asterisk (*)
  • Use MM/DD/YY format when entering dates
  • Click the search icon to find items
  • Dropdown menus show available options
Improve by: Including only process-specific information. These general Workday behaviors apply to almost every process and are assumed knowledge.

‚úÖ Process-Specific ‚Äì Perfect!

  • Closing a position is permanent and cannot be undone
  • Positions can be created ad hoc or during Create Job Requisition
  • If performance calibration is required, additional approval steps are triggered
  • Only HR Partners have permission to freeze positions

Why it works: These considerations are unique to position management processes.

üí° The "So What?" Test

Ask yourself: "Does this information apply ONLY to this specific business process, or does it apply to most Workday processes?"

If it applies broadly ‚Üí Don't include it
If it's process-specific ‚Üí Include it!

3 Optional vs. Conditional Steps

We didn't previously have the concept of optional steps clearly defined, which made them challenging to apply consistently. Let's clarify the difference to help us align on when to use each approach.

‚úì When to Use "Optional:"

Use "Optional:" ONLY when a step is truly discretionary for ALL users in ALL situations. The user may always skip this step, regardless of context or scenario.

For example, entering comments in a benefits enrollment process is truly optional in that it's not generally required under any scenario. Any user could choose to add or skip this step based on their preference.

‚ö†Ô∏è When NOT to Use "Optional:" - Conditional Steps

If a step must be completed when certain criteria are met, it is conditional, not optional. Don't label it as optional.

Instead, use these phrases:

  • "as needed" - when the user decides if it's necessary
  • "as applicable" - when specific conditions determine necessity
  • "if applicable" - when conditions may or may not apply
  • Summary of criteria - explicitly state when the step is required

For example, completing a costing questionnaire for positions requiring funding outside defaults is conditional‚Äîit's required when that specific funding scenario exists, not truly optional for all users.

‚ùå Room for Improvement - This is Conditional!

22. Optional: Complete the costing questionnaire.

Note: Only applicable for positions that require
funding outside of default organizational assignments.
Improve by: Using conditional language since this step is REQUIRED when certain criteria are met (funding outside defaults). Labeling it as optional could mislead users.

‚úÖ Better Approach - Conditional Language

22. From My Tasks, complete the costing
questionnaire as applicable.

Note: Only applicable for positions that require
funding outside of default organizational assignments.

Why it works: "As applicable" accurately conveys that this step is required under certain conditions.

4 Bullets vs. Substeps

Bullets and substeps serve different purposes. Understanding when to use each helps create clear, accurate documentation.

‚Ä¢ When to Use BULLETS

Bullets are used in two specific situations:

  1. To provide examples of possible selections or entries
  2. To indicate steps that don't explicitly need to be done (conditional sub-actions)
‚úì Bullets for Conditional Sub-Actions
1. If performance calibration is required, select Performance Calibration as the Calibration Program. If Run Once in Future is selected:
‚Ä¢ Enter Start Date.
‚Ä¢ Enter Start Time.
‚Ä¢ Verify Time Zone.

Why bullets? These sub-actions only happen IF "Run Once in Future" is selected. They're conditional, so bullets indicate they might not need to be completed. Note: Field names are still bolded because they might receive interaction.

‚úì Bullets for Examples
19. Optional: Click the (+) for additional position qualifications:
‚Ä¢ Skills
‚Ä¢ Education
‚Ä¢ Languages
‚Ä¢ Certifications
‚Ä¢ Work Experience

Why bullets? These are examples of available options. The user might select one, some, all, or none.

a) When to Use SUBSTEPS (a, b, c)

Substeps are for actions that must be completed as part of a broader step. They're not conditional‚Äîthey're required sub-actions.

Substeps can be formatted two ways:

  • Simple list: Just list the field names
  • Full sentences: Write complete action statements

Key Rule: Bold the element that receives interaction, whether it's a simple list or full sentences.

‚úì Substeps - Simple List Format
2. Enter information under the Invoice Details section for the following fields:
a. Invoice Date
b. Invoice Number
c. Invoice Amount
d. Payment Terms
e. Remit-To Address

Why substeps? These are required fields that must be filled out. Substeps indicate mandatory actions.

‚úì Substeps - Full Sentence Format
5. Complete fields under the Contract Information section:
a. Enter Contract Start Date.
b. Enter Contract End Date.
c. Select Contract Type.
d. Enter Total Contract Value.

Why substeps? Same required actions, but written as complete action statements. Both formats work‚Äîchoose based on context and clarity.

‚Ä¢ Bullets = Conditional/Examples

  • Might not need to be done
  • Depends on user choices
  • Shows available options
  • Examples of possibilities

a) Substeps = Required Actions

  • Must be completed
  • Part of broader step
  • Required fields/actions
  • Sequential sub-tasks

5 Bold Only What Receives Interaction

Our previous guidance was to bold headers, tabs, fields, and other UI elements more broadly. However, this approach made it unclear which element in a step was receiving the actual interaction. The new standard provides clearer guidance.
THE STANDARD: Within procedure steps, only bold the element that receives the interaction. Nothing else.

‚ú® Elements to Bold (When They Receive Interaction)

  • Field names when users click into them to enter data
  • Buttons when users click them
  • Checkboxes when users check/uncheck them
  • Menu items when users select them
  • Tabs when users click to navigate to them
  • Business process names when users search for and select them

‚úó Elements NOT to Bold

  • ‚úó Field names when auto-populating (no user interaction)
  • ‚úó Section names used for location context only
  • ‚úó Headers mentioned for reference
  • ‚úó Tab names mentioned in context (only bold when user selects)
  • ‚úó Page names (write in lowercase, no bold)
  • ‚úó Colons after bolded elements
  • ‚úó Punctuation at the end of bolded text

‚ùå Room for Improvement

5. Under the Contract Details section, enter
Contract Number under the Terms tab.

6. Navigate to the Employee Profile page and
update Address in the Contact
Information section.
Improve by: Bolding only the elements receiving interaction. Too much bolding makes it unclear what the user should actually click or enter.

‚úÖ Better Approach

5. Enter Contract Number under the Terms tab.

6. Enter Address in the Contact Information section.

Why it works: Only the elements receiving interaction (field names) are bolded. Clear and unambiguous.

Before bolding anything, ask: "Does the user click it or select it?"

YES ‚Üí Bold it
NO ‚Üí Don't bold it (even if it helps locate the element)

6 Always List the Action First

Start every step with what the user does, not where they do it. Action-first writing is clearer, more direct, and easier to follow.
THE PRINCIPLE: Action ‚Üí Location ‚Üí Additional Details
Always begin with the verb (Click, Enter, Select) then specify where or what.

‚ùå Location First (Room for Improvement)

‚ùå Under the Time Clock section, click Check In.

‚ùå In the Contract tab, enter Contract Number.

‚ùå From the dropdown menu, select Payment Type.

‚ùå On the Details tab, update Supplier Information.
Improve by: Starting with the action. Users have to read the entire sentence to discover what action to take.

‚úÖ Action First (Better Approach)

‚úÖ Click Check In under the Time Clock section.

‚úÖ Enter Contract Number in the Contract tab.

‚úÖ Select Payment Type from the dropdown menu.

‚úÖ Update Supplier Information on the Details tab.

Why it works: Users immediately know what to do, then where to do it.

üìç Exception: When Location Adds Essential Clarity

You can put location first ONLY when it's needed for clarity, such as:

  • Multiple elements with the same name exist on the page
  • The location is critical to finding the right element
  • Confusion is likely without location context first
Acceptable when needed for clarity:
"On the Details tab, enter Start Date."
(If there's also a Start Date field on another tab)
‚úì Examples - Action First
1. Search for and select Create Supplier Contract in the search bar.

2. Click the Supplier for the new contract.

3. Select Contract Type.

4. Enter Contract Title.

5. Select Review Invoice.

6. Click Submit to route to approvers.

7. SelectService Lines.

8. Select Recurring Invoice.

üí° Quick Rewrite Formula

Before: "In the [location], [action] [element]"

After: "[Action] [element] in the [location]"

Example transformation:
"In the dropdown menu, select Payment Terms" ‚Üí
"Select Payment Terms from the dropdown menu"

7 Better Closing Steps: Beyond "Click Submit"

Don't just end with "Click Submit." Users need to know what happens next. This small addition improves user confidence and understanding.
IMPROVED STANDARD: Always indicate what happens after submission
Two options: "to complete the process" or "to route to approvers"

‚ùå Room for Improvement

7. Click Submit.
Improve by: Adding context about what happens next. Users are left wondering: "Is that it? Does it go to someone else? Am I done?"

‚úÖ Better Approach

7. Click Submit to complete the process.

-- OR --

8. Click Submit to route to approvers.

Why it works: User knows exactly what to expect after clicking Submit.

‚úì When to Use Each Phrase

Use "to complete the process" when:

  • The task is finished after submission
  • No approval workflow follows
  • The process ends immediately
7. Click Submit to complete the process.

Use "to route to approvers" when:

  • An approval workflow begins
  • The task goes to others for review
  • Additional steps occur after submission
23. Click Submit to route to approvers.

Ask yourself: "After I click Submit, what happens?"

If the answer is "I'm done": "to complete the process"

8 Click vs. Select: Choose the Right Word

Using "click" and "select" correctly improves clarity and consistency. This aligns with Microsoft, IBM, and Workday style guides. The principle: use "select" for actions that change a persistent state (making a choice), and "click" for immediate actions without persistent state changes.

Core Principle: Use "select" when the user is making a choice from options that persists. Use "click" for immediate actions like navigation or submission.
